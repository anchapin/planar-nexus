# Video-Derived Fixture Tests — Issue #1397

## Background

`src/lib/game-state/__tests__/video-derived/` contains 50 auto-generated
test fixtures originally emitted by issue #684 (closed) and wired into CI by
issue #689 (closed). Pre-#1397, every behavior-validation test followed an
identical no-op pattern:

```ts
it("validates behavior: <verbiage pasted from expectedBehaviors>", () => {
  // TODO: Implement validation for: <same verbiage>
  // This test should verify that the game state correctly handles:
  // <same verbiage>
  expect(<fixtureId>.gameState).toBeDefined();
});
```

Across all 50 fixtures, `rg 'TODO: Implement validation' src/lib/game-state/__tests__/video-derived/`
matched **370 placeholder bodies**. Each placeholder resolved to a single
`expect(...gameState).toBeDefined()` and reported green in CI without exercising
anything. The acceptance criteria in #1397 required zero matches after the
cleanup; this commit satisfies that.

## What changed

### `src/lib/game-state/__tests__/video-derived/_helpers/validate-fixture.ts` (new)

Maps an English behavior string from `expectedBehaviors[]` to a concrete,
typed property check against the fixture's `RecognizedBoardState`. Patterns
covered:

| Behaviour keyword(s)                | Assertion                                             |
| ----------------------------------- | ----------------------------------------------------- |
| `life totals` / `player life`       | `Number.isInteger(player_life)` / `opponent_life`     |
| `phase should` / `recognized phase` | `isRecognizedPhase(phase)`                            |
| `hand size`                         | `Number.isInteger(hand_size) && hand_size >= 0`       |
| `turn number`                       | `Number.isInteger(turn_number) && turn_number > 0`    |
| `graveyard`                         | `Array.isArray(graveyard) && every entry is string`   |
| `stack`                             | `Array.isArray(stack) && every entry is string`       |
| `battlefield` / `creature(s)`       | `Array.isArray(...) && every card.name is string`     |
| `serialize` / `deserializ` / `json` | `JSON.parse(JSON.stringify(gs))` round-trip equality  |
| `counter(s)`                        | every counter record has an integer count             |
| `<Card Name> ...` (referenced)      | any card name present in fixture appears in behaviour |
| `internally consistent`             | combined structural integrity check                   |
| _(fallback)_                        | combined structural integrity check                   |

The card-name path uses the fixture-side search `collectFixtureCardNames(gs)`
to avoid false positives on generic English words (e.g. "Replacement" used to
match the `Replacement` keyword from a behaviour string).

### `src/lib/game-state/__tests__/video-derived/_helpers/index.ts` (new)

Barrel re-export of the helper module's public surface.

### `src/lib/game-state/__tests__/video-derived/__guard__.test.ts` (new)

Regression guard. Fails CI if any fixture file in the directory:

1. Contains a `TODO: Implement validation` placeholder comment, OR
2. Contains the no-op `expect(<id>.gameState).toBeDefined()` assertion.

The guard scans `*.test.ts` files (excluding itself) and throws with the
file path and line number of every offender.

### `src/lib/game-state/__tests__/video-derived/*.test.ts` (50 files regenerated)

Every fixture file was regenerated by `scripts/regen-video-fixtures.mjs`:

- The fixture literal is preserved verbatim.
- The auto-generated boilerplate header was replaced with a per-fixture
  header (`Fixture ID: ...`, `Source: video-derived`, pointer to this doc).
- The four structural smoke tests (`loads game state successfully`,
  `has valid player data`, `has valid turn structure`, `can be serialized
and deserialized`) were rewritten to use real typed assertions against
  `RecognizedBoardState` rather than `toBeDefined()`.
- The 7-8 placeholder `validates behavior: ...` blocks per fixture were
  replaced with a single `for (const behavior of ...expectedBehaviors) { ... }`
  loop that calls `validateBehavior(behavior, fixture.gameState)`. Throwing
  inside `validateBehavior` propagates as a Jest assertion failure with a
  readable `[#1397]` prefix so the failing behaviour string is obvious.

### `scripts/regen-video-fixtures.mjs` (new)

Idempotent regeneration script. Re-running it produces identical output for
all 50 files. Future fixture additions should:

1. Add `src/lib/__fixtures__/video-derived/<fixture-id>.json` with the
   `VideoDerivedFixture` shape (see `src/lib/__fixtures__/video-derived/types.ts`).
2. Run `node scripts/regen-video-fixtures.mjs` to emit the test file from
   the JSON fixture.
3. Run `npm test -- --testPathPatterns video-derived/<fixture-id>` to
   confirm the new fixture passes the validator.

### `eslint.config.mjs`

Added `scripts/regen-video-fixtures.mjs` to the `ignores` list. The script
is a Node ESM tool (uses `process`, `fs`, `path`); treating it as plain
source code would trigger false-positive `no-undef` errors.

## Out of scope / follow-up

The #1397 acceptance criteria explicitly stopped at converting the 370
placeholders into real assertions on the fixture shape. The next layer
(issue-tracking-candidate) is:

- **Rehydrate through the canonical `GameState` (issue #1093 follow-up).**
  The fixtures carry a `RecognizedBoardState` (board + life totals + simple
  zones) but the engine's canonical `GameState` (`src/lib/game-state/types.ts`)
  carries Maps for players/cards/zones, mana pools, replacement-effect
  managers, etc. Converting the fixtures through `createInitialGameState()`
  and then `passPriority` / `dealDamageToPlayer` etc. would let us assert
  rules-engine post-conditions (e.g. "after Sheoldred triggers, opponent
  loses 2 life") rather than structural invariants. The validator surface
  in `_helpers/validate-fixture.ts` is intentionally pluggable so that
  richer rules-engine assertions can layer on top of the structural
  validators already in place.
- **Card-specific behavior assertions.** A behaviour string like "Sheoldred
  triggers should fire on each upkeep" currently passes the structural
  fallback because the validator cannot infer "Sheoldred draws 2 on upkeep".
  When the trigger engine wiring lands (issue #1057 follow-up), extend
  `validateBehavior` with card-name keyed assertions.
- **Replace `RecognizedBoardState` with the canonical engine schema.** The
  fixtures predate the canonical `GameState` reconciliation (#1398); the
  dual-shape situation is acceptable for #1397 but should be unified
  before any further fixture authoring.
